# VRAM と GPU 使用量の調整

Jasna は VRAM を自動管理します。VRAM が不足すると、キュー内のフレームを
一時的にシステム RAM へ移し、また戻します。そのための設定は不要です。
以下の設定は、処理が使う VRAM と GPU 時間の量を調整するものです。

## 最大クリップサイズと時間オーバーラップ

Jasna はモザイクをクリップ — まとめて処理されるフレームの連続 — 単位で
復元します。**最大クリップサイズ**はクリップの最大長を決めます
（長いほど時間的な一貫性が向上し、VRAM を多く使います）。
**時間オーバーラップ**はクリップのつなぎ目を滑らかにします。オーバー
ラップが大きいほど処理時間が増えます。`20` を超えても通常は大きな効果は
ありません。

推奨の開始点:

- GPU が扱える範囲で最も大きい**最大クリップサイズ**を使います。
- **時間オーバーラップ**を `8` から `20` の間に設定します。
- クロスフェードは有効のままにします（処理済みのフレームを再利用する
  ため、追加コストはありません）。

限られたテストでの目安:

| 最大クリップサイズ | 時間オーバーラップ | メモ |
| -------------:| ----------------:| ----- |
| 60            | 6                | VRAM を抑えたい場合。 |
| 90            | 8                | 現在のデフォルト相当のバランス。 |
| 180           | 15               | BasicVSR++ コンパイル有効時は 12 GB 以上の VRAM が必要。無効時は少なくて済みます。 |

4K 動画はより多くの VRAM を使います。クリップサイズを下げても同程度の
品質になり、処理が速くなる場合があります。`60` 未満のクリップサイズでも
動画によっては動きますが、モデルコンパイルを無効にしてでも `60` を
推奨します。

```bash
jasna --input input.mp4 --output output.mkv --max-clip-size 90 --temporal-overlap 8 --enable-crossfade
```

## VRAM が足りない？

1. まず**最大クリップサイズ**を下げます — たとえば `180` から `60` へ。
2. それでも足りない場合は、BasicVSR++ コンパイルを無効にします（下記）。
   処理は遅くなりますが、ピーク VRAM は下がります。
3. セカンダリ復元をやめるか、より軽いものを選びます
   （[比較表](models.md)を参照）。

## 復元モデルのコンパイル

NVIDIA では、復元モデルは TensorRT エンジンにコンパイルされます。
コンパイルすると速度は上がりますが、VRAM を多く使います。パフォーマンスと
引き換えに無効化できます（AMD は常に PyTorch モデルを使います）:

```bash
jasna --input input.mp4 --output output.mkv --no-compile-basicvsrpp
```

以下はコンパイル済みエンジンのみの VRAM で、総処理 VRAM ではありません:

|                                    | Clip 60 | Clip 180 |
| ---------------------------------- | -------:| --------:|
| エンジン VRAM（コンパイルあり）     | 約 1.9 GB | 約 5.4 GB  |
| エンジン VRAM（コンパイルなし）     | 約 1.2 GB | 約 1.2 GB  |
