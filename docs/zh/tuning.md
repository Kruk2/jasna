# 调整 VRAM 和 GPU 占用

Jasna 会自动管理 VRAM: 显存不足时，排队中的帧会临时移动到系统内存，
需要时再移回来。这部分你不需要做任何配置。下面的设置控制处理占用
多少 VRAM 和 GPU 时间。

## 最大片段大小和时间重叠

Jasna 以片段为单位修复马赛克 — 片段是一起处理的一段连续帧。
**最大片段大小**限制片段的最大长度（越长 = 时间一致性越好，VRAM
占用越多）。**时间重叠**用于平滑片段衔接处的接缝；重叠越大，处理
时间越长。重叠超过 `20` 通常没有太大帮助。

推荐起点:

- 使用 GPU 能承受的最大**最大片段大小**。
- 将**时间重叠**设置在 `8` 到 `20` 之间。
- 保持交叉淡化开启（它没有额外开销 — 复用的是已经处理过的帧）。

有限测试中的参考:

| 最大片段大小 | 时间重叠 | 说明 |
| -------------:| ----------------:| ----- |
| 60            | 6                | 较低 VRAM 选择。 |
| 90            | 8                | 接近当前默认设置的平衡点。 |
| 180           | 15               | 启用 BasicVSR++ 编译时需要 12 GB+ VRAM；禁用编译时需求更低。 |

4K 视频使用更多 VRAM。较低的片段大小可能产生类似质量，并且处理更快。
低于 `60` 的片段大小在部分视频上可用，但即使需要禁用模型编译，也更
推荐 `60`。

```bash
jasna --input input.mp4 --output output.mkv --max-clip-size 90 --temporal-overlap 8 --enable-crossfade
```

## 显存不足？

1. 先降低**最大片段大小** — 例如从 `180` 降到 `60`。
2. 如果还不够，禁用 BasicVSR++ 编译（见下文）。处理会变慢，但峰值
   VRAM 会下降。
3. 跳过二级修复，或选择更轻量的方案
   （见[对比表格](models.md)）。

## 修复模型编译

在 NVIDIA 上，修复模型会编译为 TensorRT 引擎。编译会提高速度，但会
使用更多 VRAM。你可以禁用它，以性能换取更低 VRAM（AMD 始终使用
PyTorch 模型）:

```bash
jasna --input input.mp4 --output output.mkv --no-compile-basicvsrpp
```

下面仅为编译引擎本身占用的 VRAM，不是总处理 VRAM:

|                               | 片段 60 | 片段 180 |
| ----------------------------- | -------:| --------:|
| 引擎 VRAM（编译） | 约 1.9 GB | 约 5.4 GB |
| 引擎 VRAM（不编译） | 约 1.2 GB | 约 1.2 GB |
